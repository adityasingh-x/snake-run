Implementation is complete.

Review the implementation as a Staff Engineer performing a pull request review.

Before reviewing:

1. Read AGENTS.md.
2. Read plans/ACTIVE.md.
3. Read ROADMAP.md.
4. Read ARCHITECTURE.md.
5. Read any additional files required to understand the implementation.

Requirements:

- Do not implement fixes.
- Do not modify source code.
- Do not rewrite the implementation.
- Do not redesign the architecture.
- Do not propose future milestone work.
- Create a review document only.
- Save the review as reviews/IMPLEMENTATION_REVIEW.md.

Review Objectives:

1. Verify implementation matches plans/ACTIVE.md.
2. Verify alignment with AGENTS.md.
3. Verify alignment with ROADMAP.md.
4. Verify alignment with ARCHITECTURE.md.
5. Identify bugs, regressions, edge cases, or incomplete work.
6. Identify documentation inconsistencies.
7. Identify unnecessary complexity.
8. Identify scope creep beyond the approved milestone.
9. Evaluate maintainability and handoff quality.

The review document should contain:

# Executive Summary

- Overall Assessment
- Major Strengths
- Major Concerns

# Findings

For each finding:

- Severity (Critical / High / Medium / Low)
- Category (Bug / Architecture / Maintainability / Documentation / Testing / Scope)
- Description
- Recommendation

# Plan Compliance Review

Evaluate:

- Completed as planned
- Partially completed
- Missing implementation

Reference relevant sections of plans/ACTIVE.md.

# Documentation Review

Verify:

- ROADMAP.md updates
- ARCHITECTURE.md updates
- PROJECT_STATE.md updates (if required)
- Any other required documentation updates

# Testing Review

Evaluate:

- Existing tests
- Missing tests
- Verification quality

# Final Decision

Choose one:

- Approve
- Approve with Minor Changes
- Approve with Major Changes
- Reject

Favor:

- simple solutions
- maintainable code
- repository alignment
- milestone completion

Do not penalize code for lacking future features outside the approved milestone scope.
